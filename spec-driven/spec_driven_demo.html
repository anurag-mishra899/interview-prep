What is Spec-Driven Development?

A paradigm shift where specifications become executable artifacts that generate code, rather than documentation that becomes outdated. Critical for regulated industries like Wealth Management.

Why This Matters for WAM: Regulatory requirements change constantly (MiFID II, SEC rules, GDPR). When specs drive code, compliance updates become systematic regenerations, not risky manual rewrites.

Core Principles

1. Specifications Are Compliance Contracts

Not documentation that drifts. Living source files that enforce regulatory requirements through code generation.

2. Natural Language for Business Rules

Compliance officers read the same specs that generate code. No translation errors between business and tech.

3. Audit-Ready Documentation

Every feature has versioned, reviewed specs. Perfect for regulatory audits and change tracking.

4. Progressive Formalization

Match rigor to risk: Prototypes use lightweight specs, trading engines use formal specifications.

WAM-Specific Challenges & Solutions

Challenge 1: Regulatory Compliance Burden

Scenario: KYC/AML Compliance Updates

Traditional Approach: Manual code updates across 15+ services when FATF updates AML requirements. Risk of inconsistent implementation, 3-6 month delivery, extensive QA.

Spec-Driven Approach: Update compliance spec once, regenerate affected services. Systematic, consistent, audit trail built-in. 2-4 week delivery.

Benefit: 60-70% faster compliance updates, zero interpretation errors

Challenge 2: Multi-Jurisdictional Requirements

Scenario: Portfolio Reporting for US, UK, EU Clients

Traditional Approach: Separate codebases or complex conditional logic for each jurisdiction. Hard to maintain, testing nightmare, high defect risk.

Spec-Driven Approach: Jurisdiction-specific specs with shared core spec. Generate region-specific implementations from same architectural foundation.

Benefit: 50% less code, consistent business logic, easier audit

Challenge 3: Client Onboarding Complexity

Scenario: Digital Onboarding with Risk Profiling

Traditional Approach: Requirements documents disconnected from implementation. Business rules buried in code. Changes require developer + compliance officer alignment.

Spec-Driven Approach: Risk profiling questionnaire in executable spec. Compliance officer reviews spec (not code). Changes update spec, code regenerates.

Benefit: Compliance team can validate business rules directly

Challenge 4: Trade Execution & Rebalancing

Scenario: Automated Portfolio Rebalancing Engine

Traditional Approach: Complex trading logic in code. Difficult to verify correctness. Algorithm changes require extensive testing, high-risk deployments.

Spec-Driven Approach: Rebalancing algorithm defined in specification with test scenarios. Generated code includes all edge cases from spec. Contract tests validate behavior.

Benefit: Mathematical correctness verifiable, safer deployments

Challenge 5: Client Reporting & Statements

Scenario: Quarterly Performance Reports

Traditional Approach: Report templates hardcoded. Calculation logic scattered. Format changes require developer time, risk of calculation errors.

Spec-Driven Approach: Report structure and calculations in spec. Business analysts can modify reports. Calculations verified against spec.

Benefit: Business owns reporting, zero calculation drift

## Regulatory Context

**Reference:** 17 CFR 240.15l-1

## Business Rule: Risk Score Calculation

### Age Factor (30 points max)
**Rationale:** Time horizon for recovery from market downturns

| Age Range | Points | Justification |
|-----------|--------|---------------|
| < 30      | 30     | 30+ years to retirement |
| 30-50     | 20     | 10-30 years to retirement |
| > 50      | 10     | < 10 years to retirement |

**Regulatory Basis:** FINRA Notice 12-25 (time horizon consideration)

### Income Factor (25 points max)
**Rationale:** Capacity to absorb losses

| Annual Income | Points | Justification |
|---------------|--------|---------------|
| > $500k       | 25     | High loss capacity |
| $200k-$500k   | 15     | Moderate capacity |
| < $200k       | 5      | Limited capacity |

**Regulatory Basis:** SEC Investment Advisor Act Section 206

### Investment Horizon (30 points max)
| Time Horizon  | Points | Justification |
|---------------|--------|---------------|
| > 10 years    | 30     | Long-term growth focus |
| 5-10 years    | 20     | Medium-term planning |
| < 5 years     | 10     | Short-term needs |

### Risk Classification Thresholds
| Score Range | Classification | Asset Allocation Guideline |
|-------------|----------------|----------------------------|
| 70-100      | AGGRESSIVE     | 80-100% equities |
| 50-69       | MODERATE       | 50-70% equities |
| 0-49        | CONSERVATIVE   | 20-40% equities |

**Compliance Note:** Classifications align with model portfolios reviewed by CIO

## Acceptance Criteria

**GIVEN** client age 35, income $300k, horizon 15 years
**WHEN** risk assessment calculated
**THEN** score = 20 + 15 + 30 = 65
**AND** classification = MODERATE
**AND** recommendation = "50-70% equity allocation"

## Audit Requirements
- Every calculation must log inputs, score, and classification
- Retain records for 7 years (SEC recordkeeping rule)
- Flag manual overrides for compliance review

// AUTO-GENERATED from risk-profiling/SPEC.md
// DO NOT EDIT - Update spec and regenerate

import { AuditLogger } from '@shared/compliance';

interface RiskAssessment {
  clientId: string;
  age: number;
  income: number;
  timeHorizon: number;
}

interface RiskResult {
  score: number;
  classification: 'AGGRESSIVE' | 'MODERATE' | 'CONSERVATIVE';
  recommendation: string;
  regulatoryBasis: string;
  auditId: string;
}

/**
 * Calculate client risk score per SEC Reg BI requirements
 * Spec: specs/risk-profiling/SPEC.md v2.1
 * Regulatory: 17 CFR 240.15l-1
 */
export async function calculateRiskScore(
  assessment: RiskAssessment
): Promise {

  // Age Factor (max 30 points) - FINRA Notice 12-25
  const ageScore =
    assessment.age < 30 ? 30 :
    assessment.age < 50 ? 20 : 10;

  // Income Factor (max 25 points) - SEC IA Act Section 206
  const incomeScore =
    assessment.income > 500000 ? 25 :
    assessment.income > 200000 ? 15 : 5;

  // Time Horizon (max 30 points)
  const horizonScore =
    assessment.timeHorizon > 10 ? 30 :
    assessment.timeHorizon > 5 ? 20 : 10;

  const totalScore = ageScore + incomeScore + horizonScore;

  // Classification per spec thresholds
  const classification =
    totalScore > 70 ? 'AGGRESSIVE' :
    totalScore > 50 ? 'MODERATE' : 'CONSERVATIVE';

  const recommendation =
    classification === 'AGGRESSIVE' ? '80-100% equity allocation' :
    classification === 'MODERATE' ? '50-70% equity allocation' :
    '20-40% equity allocation';

  // Audit trail (SEC recordkeeping requirement)
  const auditId = await AuditLogger.log({
    action: 'RISK_ASSESSMENT',
    clientId: assessment.clientId,
    inputs: assessment,
    outputs: { totalScore, classification },
    regulatoryBasis: '17 CFR 240.15l-1',
    timestamp: new Date(),
    retentionYears: 7
  });

  return {
    score: totalScore,
    classification,
    recommendation,
    regulatoryBasis: 'SEC Reg BI - Suitability Assessment',
    auditId
  };
}
